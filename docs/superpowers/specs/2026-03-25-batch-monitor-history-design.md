**Date:** 2026-03-25
**Scope:** Desktop app (Tauri + HTML/CSS/JS frontend + Rust backend)

## 1. History Fixes

### Bug 1 — Stale history after test run
After `runTest()` completes (success or failure), call `await loadHistory()` so the History page immediately reflects the new entry.

### Bug 2 — Wrong logs on expand when filter is active
**Fix:** In `refreshHistoryTable()`, set `data-id="${h.id}"` on each `<tr>`. Change `toggleHistoryDetail(btn, index)` signature to `toggleHistoryDetail(btn, id)`. Look up entry via `historyData.find(h => h.id === id)`.

**ID origin:** UUID v4 string (`uuid::Uuid::new_v4().to_string()`) generated by Rust backend at history entry creation. Always present in current code.

**Migration for old entries without id:**
In `loadHistory()`, after fetching from backend, assign `crypto.randomUUID()` to any entry missing `id`. Then persist the migrated array back to the backend via a new `update_history_ids(entries)` Tauri command **or** simply call the existing `clear_history` + re-save — but the simplest correct approach is: since history is stored by the Rust backend as JSON, and the Rust `HistoryEntry` struct has always included `id` (it was there from the start), old entries will also have `id`. The migration fallback is therefore a defensive no-op in practice. Still guard with: `if (!h.id) h.id = crypto.randomUUID();` — ephemeral only (not persisted), since no code caches these IDs across reloads.

## 2. Batch Tab

### Struct Reference
`SmtpFormInput` fields (from `lib.rs`): `host, port, encryption, username, password, from, to, subject, body, html_body, timeout_secs`
`SmtpTestOutput` fields (from `lib.rs`): `success, message, logs (Vec<LogEntry>), elapsed_ms, diagnostics`

`subject` **is** a field in `SmtpFormInput`. No backend changes needed for batch.

### UI
- **Toolbar:** Add Row | Run All | Clear | Export ▾ (CSV / JSON)
- **Job table columns:** #, Host, Port, Encryption, Username, Password, From, To, Subject, Status, Latency, Expand (chevron)
- All data cells use `<input>` elements inside `<td>` (not contenteditable — simpler focus handling)
- **Add Row defaults:** Port=587, Encryption=starttls, all others empty
- **Status column per row:** grey dot (idle) → spinner (running) → green badge "OK" / red badge "FAIL" with hover title showing error message
- **Expand chevron:** toggles inline log drawer beneath the row (same pattern as history detail)
- **Summary bar** (shown only after all jobs finish): Total X | Passed Y | Failed Z | Avg Latency N ms

### Validation
Before dispatching a row, check `host`, `from`, `to` are non-empty. If any are blank, mark that row FAIL immediately with message "Missing required fields (host, from, to)" — do not dispatch to backend.

### Concurrency
Sliding window of max **10 concurrent** jobs. Jobs start in row-index order (row 0 first). When any job completes (success or fail, including validation fail), its slot is immediately freed and the next pending row starts. Implemented as a queue:
```js
async function runWithConcurrency(jobs, limit) {
  const queue = [...jobs];
  const active = new Set();
  // ... standard semaphore pattern
}
```

### Re-run behavior
Clicking Run All when results already exist resets all row statuses to idle first, clears all log drawers, then starts a fresh run.

### Export
- **CSV:** `host,port,encryption,from,to,subject,status,latency_ms,timestamp`
  - `status`: `"success"` or `"failed"` or `"skipped"` (validation fail)
  - `timestamp`: ISO 8601 UTC (e.g. `2026-03-25T14:32:00.000Z`)
  - Mid-run export: in-flight rows export with status `"running"`, latency `""`
- **JSON:** `[{ host, port, encryption, from, to, subject, success, message, elapsed_ms, logs, exported_at }]`

### Error serialization from invoke()
Tauri surfaces Rust `Err(String)` as a JS exception. Catch pattern:
```js
try {
  result = await invoke("smtp_test", { input });
} catch (err) {
  // err is a plain string from Tauri when Rust returns Err(String)
  errorMessage = typeof err === "string" ? err : JSON.stringify(err);
}
```

## 3. Monitor Tab

### Scope
Connectivity only: TCP connect → read banner → send EHLO → read response → disconnect. Status "REACHABLE" (not "Healthy") to make scope clear. Not an auth or delivery check.

### Status derivation rule
```
reachable=false              → DOWN
reachable=true, ms >= 2000   → SLOW
reachable=true, ms < 2000    → REACHABLE
interval tick running        → CHECKING (transient)
```
SLOW threshold: **2000 ms**, fixed in v1, not configurable.

### UI
- **Toolbar:** Add Server | Interval: [30s | 1m | 5m | 10m] | Start All | Stop All
- **Server cards grid (2-column):** Each card shows:
  - Server name + `host:port`
  - Status badge: REACHABLE (green) / DOWN (red) / SLOW (yellow) / CHECKING (spinner)
  - Last response time (ms), or "—" before first check
  - Sparkline (Chart.js 4.4.0, last 20 readings, 120×40 px)
  - Uptime %: `(reachable_checks / total_checks * 100).toFixed(1)%`, or "—" before first check
  - Last checked: relative time ("12s ago"), updated every second by a shared display-refresh `setInterval(updateRelativeTimes, 1000)`
  - Per-card controls: Start / Stop button (toggles that card's interval), Delete button
- **Card click (body area):** toggles an inline detail panel below the card showing last 50 check results (timestamp, latency ms, status string)

### Per-card Start/Stop
Each card has its own running state. Start All / Stop All toggle all cards. A card's individual Start/Stop only affects that card.

### Notification behavior
- Request permission once on first "Start All" click.
- If `Notification.permission === "denied"`: show info toast once, set a `notificationsDenied` flag in memory, never request again.
- Fire notification only on state **transition** (previous state ≠ new state). Transitions that trigger: any→DOWN, DOWN→REACHABLE, any→SLOW, SLOW→REACHABLE.
- If `window.Notification` is undefined (sandboxed env): skip silently.

### localStorage
- **Key:** `smtplab-monitor-servers`
- **Schema:**
```json
[{ "id": "<uuid>", "name": "My Server", "host": "smtp.gmail.com", "port": 587, "interval_ms": 60000 }]
```
- Written on: add server, delete server, edit server name/host/port/interval (on input blur).
- Runtime data (sparkline buffer `number[]`, uptime counts `{ok, total}`, last check timestamp) is **in-memory only**.

### Backend — `smtp_ping` command

```rust
#[derive(Serialize)]
pub struct PingResult {
    pub reachable: bool,
    pub latency_ms: u64,
    pub banner: String,
    pub error: Option<String>,
}

#[tauri::command]
async fn smtp_ping(host: String, port: u16) -> Result<PingResult, String>
```

Implementation:
1. Resolve `format!("{host}:{port}")` via `ToSocketAddrs`
2. **Start timer** (`Instant::now()`)
3. `TcpStream::connect_timeout(&addr, Duration::from_secs(5))`
4. Set `read_timeout` and `write_timeout` to `Some(Duration::from_secs(5))` on the stream
5. Read banner line (up to 512 bytes, stop at `\n`)
6. Write `b"EHLO smtplab\r\n"`
7. Read response line (up to 512 bytes, stop at `\n`)
8. **Stop timer** → `latency_ms = start.elapsed().as_millis() as u64` (covers connect + banner + EHLO response; excludes QUIT)
9. Send `b"QUIT\r\n"` and call `shutdown(Shutdown::Both)` to cleanly close
10. Return `PingResult { reachable: true, latency_ms, banner: banner_line.trim().to_string(), error: None }`
11. On any error at any step: return `PingResult { reachable: false, latency_ms: 0, banner: String::new(), error: Some(e.to_string()) }`

**Registration:** `smtp_ping` **must** be added to `tauri::generate_handler![..., smtp_ping]` in the `run()` function. Omitting this causes a silent runtime failure — `invoke('smtp_ping')` will reject without a compile error.

## 4. Chart.js Bundling

- **Version:** `4.4.0` (pinned)
- **File:** `desktop-app/src/chart.umd.min.js`
- **Obtain:**
  ```bash
  curl -L -o "desktop-app/src/chart.umd.min.js" \
    "https://cdn.jsdelivr.net/npm/chart.js@4.4.0/dist/chart.umd.min.js"
  ```
- **HTML:** `<script src="chart.umd.min.js"></script>` before `app.js` — exposes `window.Chart`
- **Sparkline config:**
  ```js
  new Chart(canvas, {
    type: 'line',
    data: { labels: Array(20).fill(''), datasets: [{ data: [], borderColor: color, tension: 0.4, fill: false, pointRadius: 0 }] },
    options: { responsive: false, animation: false, plugins: { legend: { display: false } }, scales: { x: { display: false }, y: { display: false } } }
  })
  ```
- **Fallback:** If `window.Chart` is undefined, render `<span class="chart-unavailable">(chart unavailable)</span>` in place of canvas.

### CSP note
Tauri v2 default CSP allows local file loads. The `chart.umd.min.js` file is served from `frontendDist` alongside `index.html`, so no CSP changes are needed. Verify `tauri.conf.json` has no restrictive `script-src` override.

## 6. Error Handling Summary

| Area | Scenario | Behaviour |
|------|----------|-----------|
| Batch | Row missing host/from/to | Mark FAIL immediately, skip invoke |
| Batch | invoke("smtp_test") rejects | Catch error string, show in log drawer, mark FAIL |
| Batch | Export mid-run | in-flight rows export with status "running", latency "" |
| Monitor | smtp_ping returns reachable=false | Show DOWN, log error string in detail panel |
| Monitor | invoke("smtp_ping") rejects | Treat as reachable=false with error message |
| Monitor | Notification permission denied | Show toast once, set flag, skip future notifications |
| Monitor | window.Chart undefined | Show "(chart unavailable)" text, no crash |
| History | Entry missing id (old data) | Assign ephemeral crypto.randomUUID() in JS, not persisted |
